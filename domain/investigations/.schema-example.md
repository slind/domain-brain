## Schema Documentation

This file documents the investigation node schema. All investigation nodes follow this structure:

### Frontmatter Fields

- **id** (string, required): Unique node identifier in format `{PREFIX}-{NNN}` (e.g., SEC-001, API-003)
  - User provides prefix at capture time (defaults to INV if not specified)
  - Sequence number auto-assigned by scanning existing nodes with same prefix

- **status** (enum, required): Current lifecycle state
  - `open` — active, being worked
  - `blocked` — waiting on initiative completion or external input
  - `resolved` — concluded; Resolution section filled

- **expanded** (boolean, required): Whether this node has been through analyze-issue
  - Set to `false` on creation
  - Set to `true` after first `domain:investigation:analyze-issue` session completes
  - Used by query command to identify nodes that haven't been expanded yet

- **opened** (date, required): Date this investigation node was created (YYYY-MM-DD format)

- **parents** (array of strings, required): List of parent node IDs
  - Empty array `[]` indicates a root node
  - Can contain multiple parent IDs (nodes can be child of multiple investigations)
  - Format: `["SEC-002", "API-001"]`

### Body Sections

All sections use markdown heading level 2 (##).

#### Question (H1 title)
The investigation question or topic. This is the H1 title of the document.

#### Initiatives
Checklist of actions needed to collect information for resolution.
- Format: markdown checklist (`- [ ] item` or `- [x] item`)
- Items can be added during capture or analyze-issue sessions
- Items can be promoted to child nodes during analyze-issue

#### Evidence
Two subsections: "Distilled Knowledge" and "Provisional Leads"

**Distilled Knowledge** — links to validated domain knowledge
- Format: `[distilled] <claim> (<source-file>:<line-number>)`
- Example: `[distilled] Requirements specify OAuth 2.0 support (requirements.md:42)`
- Added automatically by /refine investigation-linking sub-agent
- Can be added manually via domain:investigation:capture

**Provisional Leads** — unvalidated information
- Format: `[lead] <information>`
- Example: `[lead] Team discussion suggested JWT tokens might be sufficient`
- Added via domain:investigation:capture
- Never appears in /query responses (only in domain:investigation:query)

#### Child Nodes
Informational section showing child investigations.
- Populated by scanning all investigation files for parent references
- Not edited directly — discovered from other nodes' `parents` frontmatter
- Lists child node IDs and their questions

#### Resolution
Filled when node status changes to `resolved`.
- Contains: conclusion summary
- References to ADRs created during resolution (with ADR IDs)
- Date resolved
- Created by `domain:investigation:resolve` command
