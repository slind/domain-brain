# Research: Domain README — Consolidate Command

**Feature**: 010-onboard-tour
**Date**: 2026-03-18

## Summary

No external research required. All technical decisions are resolved by existing patterns in the Domain Brain codebase and constitution.

## Resolved Decisions

### Decision 1: Output file location

**Decision**: `domain/README.md` (relative to the domain brain root, alongside `config/` and `distilled/`)

**Rationale**: Keeps the README co-located with the domain it describes. Consistent with the domain root directory layout established by ADR-006. Discoverable by anyone navigating the domain directory.

**Alternatives considered**:
- Repository root `README.md`: Rejected — multiple domain brains may coexist in one repo (ADR-006); a repo-root README cannot represent multiple domains.
- `distilled/README.md`: Rejected — the README is not a distilled knowledge entry; placing it in `distilled/` would mix output formats.

---

### Decision 2: Overwrite vs. append strategy

**Decision**: Full overwrite on every `/consolidate` run.

**Rationale**: The README is a generated snapshot, not an append-only log. Appending would accumulate stale content over time, defeating the purpose of the document.

**Alternatives considered**:
- Append with timestamp header: Rejected — produces a multi-snapshot file that is difficult to read and shares no structure with a conventional README.
- Diff-aware update (only changed sections): Rejected — adds unnecessary complexity; a full overwrite is simpler and safer given the small file size.

---

### Decision 3: Top priorities cap

**Decision**: Top 5 open items, ordered high → medium → low.

**Rationale**: 5 items fits on a single terminal screen and is consistent with the display conventions used elsewhere in the system (e.g., triage backlog display caps groups at 7). A fixed cap avoids requiring a config file for v1.

**Alternatives considered**:
- Configurable `--top N` flag: Deferred to v2 — adds implementation complexity for marginal v1 benefit.
- All open items: Rejected — README becomes unreadably long for mature domains.

---

### Decision 4: Intended Usage section content

**Decision**: Static prose generated by the command (not sourced from a file), describing the four primary commands and the capture → refine → query workflow.

**Rationale**: The usage description is about Domain Brain itself, not about the specific domain — it does not vary between domains and does not need to be queryable or governed. Static prose embedded in the command is the simplest approach consistent with the Extension-First principle.

**Alternatives considered**:
- Sourced from a `config/usage.md` file: Rejected — adds a new config file with no clear governance mechanism and no benefit over static prose.

---

### Decision 5: Changelog entry format

**Decision**: Append a standard changelog entry to `distilled/changelog.md` on every `/consolidate` run, using the format:

```markdown
## YYYY-MM-DD — Consolidate Session

### Generated
- [consolidate]: domain/README.md → created/updated (N interfaces, M priorities)

---
```

**Rationale**: Consistent with the changelog format used by `/refine` and `/consistency-check`. Makes consolidate runs auditable in the same log used for all other domain brain operations.
